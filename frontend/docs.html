Relora Documentation

Everything you need to set up Relora, connect your webhook sources, and keep deliveries running reliably.

What is Relora?

Relora is a self-hosted webhook relay. Instead of your app receiving webhooks directly from Stripe, GitHub, or Shopify, those services send to Relora first. Relora then delivers the event to your app — with retries, failure tracking, and replay built in.

The problem it solves: Webhooks are fire-and-forget. If your server is down, restarting, or slow when a webhook arrives, the event is lost. Relora accepts the event immediately (so the sender gets a fast 200 OK), persists it to Postgres, and keeps retrying delivery to your app until it succeeds.

What you get:

  • Every event saved before your app has to handle it
  • Automatic retries with exponential backoff
  • A dashboard showing every delivery attempt and why it failed
  • One-click replay for any event or time window
  • Slack/email alerts when deliveries fail

Quickstart

You need Docker and Docker Compose installed. The whole stack runs in a single command.

1. Clone the repo

git clone https://github.com/yourname/relora.git
cd relora

2. Configure environment

cp .env.example .env

Open .env and set at minimum:

POSTGRES_PASSWORD=choose-a-strong-password
SECRET_KEY=a-long-random-string-for-jwt-signing

3. Start the stack

docker compose up -d

This starts Postgres, the API server, the delivery worker, and Nginx. Wait about 10 seconds for everything to be healthy.

Prefer a managed platform? You can also deploy to Railway, Render, or Fly.io using their Docker Compose support. Set the same environment variables and the stack starts identically.

4. Open the dashboard

Go to http://localhost and create your account. Then follow the setup steps shown in the dashboard.

That's it. You now have a working webhook relay. The next step is adding a destination (your app's endpoint) and pointing your webhook source at Relora's ingest URL.

How it works

Every webhook goes through three stages:

  1. Ingest — Your source (Stripe, GitHub, etc.) sends a POST to Relora's ingest URL. Relora responds with 200 OK immediately and saves the event to Postgres. Your source never waits for your app to respond.
  2. Delivery — A background worker picks up the event and POSTs it to your configured destination (your app). If delivery fails, it retries automatically with exponential backoff.
  3. Recovery — If all retries are exhausted, the event appears in Failed Deliveries. You can investigate why it failed and replay it once you've fixed the issue.

Adding destinations

A destination is the URL where Relora should deliver events — typically your app's webhook endpoint.

  1. Go to Destinations in the sidebar and click Add destination.
  2. Enter a name (e.g. "Production API") and the URL of your endpoint (e.g. https://api.yourapp.com/webhooks).
  3. Click Save.

You can configure optional settings for each destination:

SettingWhat it does
Max retriesHow many times to retry before giving up. Default: 5.
Backoff base (seconds)Starting delay between retries. Doubles each attempt. Default: 2s → 4s → 8s → 16s → 32s.
Reliability goal (%)The success rate you expect. Relora will warn you if delivery falls below this threshold.
Custom headersExtra headers to send with every delivery, e.g. Authorization: Bearer your-token.

Destination health

Each destination shows one of three health states:

  • Healthy — All deliveries are passing through normally.
  • Recovering — The destination recently failed. Relora is sending test deliveries to check if it has recovered. No action needed — it will return to Healthy automatically.
  • Suspended — Too many consecutive failures. Deliveries are paused. Fix the underlying issue with your endpoint, then click Reset to resume.

Connecting sources

Your ingest URL is shown in the dashboard under Settings → API Credentials. It looks like:

http://your-server/api/v1/ingest

You'll also need your API key from Settings. Include it in every request as the X-Relora-API-Key header.

Stripe

  1. Go to Stripe Dashboard → Developers → Webhooks.
  2. Click Add endpoint.
  3. Set the endpoint URL to your Relora ingest URL.
  4. Select the events you want to receive.
  5. In Relora Settings, go to Source Secrets → Stripe and paste your Stripe webhook signing secret. Relora will verify every Stripe request is genuine.

GitHub

  1. Go to your repository or organization Settings → Webhooks → Add webhook.
  2. Set the Payload URL to your Relora ingest URL.
  3. Set Content type to application/json.
  4. Optionally set a secret and paste it into Relora Settings → Source Secrets → GitHub.

Shopify

  1. Go to Shopify Admin → Settings → Notifications.
  2. Under Webhooks, click Create webhook.
  3. Set the URL to your Relora ingest URL and paste your Shopify secret into Relora Settings.

Any other source

Any service that can send a POST request can use Relora. Include the API key header:

curl -X POST http://your-server/api/v1/ingest \
  -H "Content-Type: application/json" \
  -H "X-Relora-API-Key: hk_live_your_api_key" \
  -d '{"event_type": "order.placed", "data": {"order_id": "ord_123"}}'

Setting up alerts

Relora can notify you when deliveries fail. Go to Alerts and click Add alert.

Slack

Create an Incoming Webhook in your Slack workspace and paste the URL into the alert configuration.

Email (SMTP)

Enter your SMTP server details. Most teams use SendGrid, Mailgun, or their own SMTP server.

Trigger conditions

  • Failed deliveries ≥ N — Fire when your Failed Deliveries queue reaches a certain depth. Useful for catching bursts of failures.
  • Success rate < N% — Fire when delivery success drops below your threshold over the last 24 hours.

Dashboard — Overview

The Overview page is your homepage. It shows:

  • Activity feed — Every delivery attempt in real time, as events happen.
  • Needs Attention — Appears only when something requires action: failed deliveries, suspended destinations, or missed reliability goals.
  • Destination health — Quick view of all your destinations and their current status.

If everything is healthy, the Overview is quiet. That's intentional — you should only need it when something is wrong.

Failed Deliveries

Events appear here when they could not be delivered after all retry attempts. For each event you can see:

  • The exact error that caused the failure (e.g. "Connection refused", "HTTP 500", "SSL certificate expired")
  • The full delivery attempt history with timestamps and response bodies
  • A plain-English explanation of what the error means and how to fix it

Once you've fixed the issue on your destination, replay the event with the Replay button.

Bulk Replay

Use Bulk Replay after fixing a broken destination to re-send all events from a time window. For example, if your staging server was down from 2pm–4pm and you want to replay everything that was missed:

  1. Go to Bulk Replay in the sidebar.
  2. Set the From and To times to the outage window.
  3. Optionally select a specific destination (or leave as "All destinations").
  4. Set the rate limit (events per minute) to avoid overwhelming your endpoint.
  5. Click Start replay.

Relora will re-queue every event from that window and deliver them at the rate you set. You can watch progress in real time.

Reliability Trends

Shows delivery success rates and response times per destination over the last 7 days. Use this to:

  • See which destinations are consistently slow or unreliable
  • Spot when a problem started
  • Confirm a fix is working after an incident

API reference

All API calls require your API key as X-Relora-API-Key: hk_live_your_key.

Ingest an event

POST /api/v1/ingest

Body (JSON):
{
  "event_type": "payment.succeeded",   // required — any string you choose
  "data": { ... },                      // required — the event payload
  "event_id": "evt_unique_id",          // optional — for deduplication
  "ordering_key": "user_123"            // optional — ensures FIFO per key
}

Returns 200 OK with {"webhook_id": "...", "status": "accepted"}. The event is saved before this response is sent.

List recent events

GET /api/v1/webhooks?limit=50&status=failed&search=payment

Get event details

GET /api/v1/webhooks/{webhook_id}

Returns the event payload, all delivery attempts, and the current status.

Replay an event

POST /api/v1/webhooks/{webhook_id}/replay

Replay a time window

POST /api/v1/webhooks/replay-window

Body:
{
  "from_time": "2025-01-15T14:00:00Z",
  "to_time":   "2025-01-15T16:00:00Z",
  "destination_id": "optional-uuid",
  "replay_rate_per_minute": 100
}

FAQ

What happens if my server is down when a webhook arrives?

Relora accepts the event immediately and stores it. Delivery retries will continue for as long as your retry policy allows (default: 5 attempts over ~60 seconds). If all retries fail, the event goes to Failed Deliveries where you can replay it once your server is back up.

Can I receive webhooks from multiple sources?

Yes. All sources point at the same ingest URL. You can use the event_type field to distinguish them, and set up multiple destinations with different filter rules if you want to route events to different endpoints.

How do I verify webhook signatures?

Go to Settings → Source Secrets and add your signing secrets for Stripe, GitHub, or Shopify. Relora will verify every incoming request and reject ones with invalid signatures.

Is there a rate limit on ingestion?

Relora uses a token bucket rate limiter per API key. The default is generous for most use cases. You can adjust limits in the environment configuration.

How do I update Relora?

git pull
docker compose down
docker compose up -d --build

Migrations run automatically on startup.

operational guarantees

Everything production needs.

Each feature exists because webhooks fail in the real world. Relora handles the failure cases your application shouldn't have to.

01
FIFO_DELIVERY per ordering key
Events delivered in strict order per customer or ordering key, even under concurrent load. Implemented with SELECT FOR UPDATE SKIP LOCKED. No external message broker required.
02
EXP_RETRY with Retry-After support
Exponential backoff: 30s → 1m → 2m → 4m → … Respects Retry-After headers. Configurable max attempts per destination. Every attempt logged with status code, response body, and latency.
03
DEAD_LETTER_QUEUE 14 failure categories
Failed events captured and classified into 14 categories: TIMEOUT, NETWORK_ERROR, AUTH_FAILURE, RATE_LIMITING, and more. 0–100 health score per destination. Grouped into incidents with recommended recovery actions.
04
CIRCUIT_BREAKER per destination
Per-destination state machine: OPEN / HALF-OPEN / CLOSED. Broken endpoints suspend automatically — no thundering herd against a degraded service. Auto-recovery when the destination comes back online.
05
BULK_REPLAY rate-limited recovery
Re-deliver any event or time window. Rate-limited to protect downstream services during recovery. Individual events replayable from the webhook detail view. Full tamper-evident audit trail of every action.
06
FAN_OUT HMAC signing + multi-tenant
Single event to multiple destinations from one ingest. Standard Webhooks HMAC signing built in. Multi-tenant: projects, team roles, per-project API keys. One self-hosted deployment serves multiple teams.
architecture

Instant accept. Reliable deliver.

Relora sits between your event sources and your endpoints. Returns 200 immediately — delivery, retries, and failure handling happen asynchronously.

01 // RECEIVE
Instant Accept
Stripe, GitHub, Shopify, or your own app POSTs to Relora. Returns 200 OK immediately and persists the event to Postgres. Your sender is never blocked waiting for your application.
02 // RELAY
Async Retry
Background workers deliver events to your destinations. On failure, exponential backoff kicks in. Circuit breakers protect degraded endpoints from thundering-herd. Every attempt is logged.
03 // RECOVER
Full Visibility
Every attempt logged with status code and latency. Dashboard shows DLQ depth, circuit states, and delivery trends. Replay any time window in one click.

faq

Common questions.

What is a webhook? +

A webhook is an automatic HTTP notification sent by one service to another when something happens. Stripe sends one when a payment succeeds. GitHub sends one when code is pushed. Your app receives these and reacts — updating a database, triggering a build, sending an email.

Why not send webhooks directly to my app? +

Your server can be temporarily down, slow to respond, or rate-limited. Most webhook senders give up quickly and don't retry reliably. Without a relay layer, those events are lost forever. Relora accepts every event immediately (always 200), then guarantees delivery through retries, queuing, and dead letter capture — so your application is never the weak link.

What happens when delivery fails? +

Relora retries with exponential backoff: 30 seconds, then 1 min, 2 min, and so on — respecting Retry-After headers when present. Every attempt is logged with status code, response body, and latency. After all retries are exhausted the event moves to the Dead Letter Queue, where it is classified and held for replay.

How does replay work? +

In the dashboard go to Bulk Replay, pick a time window and destination, and start the job. Relora re-queues all events from that window through the full delivery pipeline at a rate you control — so a recovering downstream isn't overwhelmed. Individual events can also be replayed directly from the webhook detail view.

What reliability guarantees exist? +

FIFO delivery per ordering key using SELECT FOR UPDATE SKIP LOCKED — no events are lost or duplicated under concurrent workers. Circuit breakers prevent thundering-herd on degraded endpoints. The DLQ captures every permanently-failed event with a 0–100 health score and automatic failure classification. Postgres is the single source of truth; no external message broker required.

At-least-once or exactly-once delivery? +

At-least-once. Under normal operation each event is delivered exactly once. Under retries, a destination may receive the same event more than once. To achieve exactly-once semantics, send an idempotency key with each event — Relora deduplicates events with the same key before delivery.

How does the Dead Letter Queue work? +

When an event exhausts all retries it moves to the DLQ. Relora classifies the failure into one of 14 categories (AUTHENTICATION, NETWORK, TIMEOUT, SERVER_ERROR, RATE_LIMITING…), groups related failures into incidents, and computes a 0–100 health score. The DLQ Intelligence dashboard shows failure breakdown, growth trend, top failing destinations, and recommended actions.
